# 遇见路得 MVP 收口设计

日期：2026-06-29

## 目标

遇见路得的第一版目标是跑通一个真实可验收的社区婚配闭环，而不是继续扩展功能面。产品主线为：

注册登录 -> 完善资料与信仰档案 -> 信仰测试 -> 提交背书并由管理员审核 -> 完成恋爱必修课 -> 进入匹配池 -> 匹配与互相表达意向 -> 自动开通私聊 -> 在全站广场进行基础互动。

本轮收口采用“闭环优先”方案。优先修正路由、部署、数据状态和关键权限，让已有前后端代码形成一致的产品路径。

## 已确认决策

| 项目 | 决策 |
|---|---|
| 产品名称 | 遇见路得 |
| 首页路径 | `/` |
| 应用区路径 | `/app` |
| API 路径 | `/api/*` |
| MVP 主线 | 完整入池、匹配、私聊、广场互动闭环 |
| 背书审核 | 用户提交牧者或成熟引荐人，管理员手动通过或拒绝 |
| 社群 MVP | 全站广场；发帖、评论、点赞 |
| 关系确立生命周期 | 暂不纳入 MVP 验收 |
| 牧者台 | 暂不暴露为 MVP 主入口 |
| 旧 `matcha/` | 作为参考或遗留项目，不作为当前主线 |

## 路由与部署结构

| 路径 | 用途 |
|---|---|
| `/` | 遇见路得宣传首页，保留当前根目录静态首页 |
| `/app` | React 登录后应用区 |
| `/app/login` | 登录页 |
| `/app/register` | 注册页 |
| `/api/*` | Express 后端 API |

实现收口方向：

| 文件或配置 | 调整方向 |
|---|---|
| `index.html` | 所有进入应用的按钮统一指向 `/app` 或 `/app/register` |
| `web/vite.config.js` | `base` 改为 `/app/` |
| `web/src/main.jsx` | `BrowserRouter basename` 改为 `/app` |
| `web/src/api/client.js` | API base 统一请求 `/api` |
| `server/src/index.js` | `/` 服务静态首页，`/app` 服务 `web-dist`，`/app/*` 做 SPA fallback |

这样可以让首页、应用区和 API 形成清晰边界，也方便后续使用 nginx 或其他反向代理部署。

## MVP 功能闭环

| 阶段 | 用户动作 | 系统结果 |
|---|---|---|
| 注册登录 | 邮箱与密码注册或登录 | 进入个人中心 |
| 完善资料 | 填写个人资料和信仰档案 | 更新资料完成度与入池状态 |
| 信仰测试 | 完成 20 题，答对 15 题通过 | 点亮信仰测试条件 |
| 背书审核 | 提交牧者或成熟引荐人信息 | 管理员手动审核为通过或拒绝 |
| 恋爱必修课 | 完成轻量课程单元 | 点亮课程入池条件 |
| 进入匹配池 | 五项条件全部满足 | 开放候选人、表达意向、发帖权限 |
| 匹配私聊 | 双方互相表达意向 | 自动创建私聊通道 |
| 社群广场 | 入池用户发帖、评论、点赞 | 未入池用户只能浏览 |

不纳入第一版验收的能力：

- 支付和真实订单。
- 真实 AI RAG 咨询。
- 邮箱验证自动流程。
- 牧者或引荐人邮件确认链接。
- 小组、举报、收藏、关注、活动和复杂通知。
- 关系确立后的双方牧者审核流程。

## 数据模型与后端修正

后端不重做架构，只修正会阻断 MVP 闭环的断点。

| 问题 | 修正方向 |
|---|---|
| `match_status` 缺少 `matched` | 在 schema 中加入 `matched`，用于双方互相表达意向后开通私聊 |
| `match_status` 缺少 `in_relationship` | 关系确立流程后置，MVP 不写入该状态 |
| `relationships.routes.js` 依赖不存在的 `endorser_id` | 该路由不作为 MVP 主流程；避免影响启动，后续单独设计牧者审批模型 |
| 管理员审核背书 | 保留 `endorsements.state = pending/verified/rejected`，管理台调用审核接口 |
| 入池判断 | 统一由 `match-qualification.js` 判断五项条件 |
| 课程奖励 | 恋爱必修课只作为入池门槛；凯勒课才发徽章、曝光翻倍和 VIP 体验 |
| 私聊开通 | 双方 `intent_sent` 后创建 `chat_channels`，匹配状态进入 `matched` |
| 社群发帖门槛 | 发帖前复用入池判断；未入池只能读广场 |

入池五项条件为：

1. 个人资料完整且同意隐私授权。
2. 信仰档案完整。
3. 信仰基础测试通过。
4. 至少一个牧者或成熟引荐人背书已验证。
5. 恋爱必修课完成。

## 前端页面与权限状态

| 页面 | MVP 要求 |
|---|---|
| `/app` | 个人中心，显示五项入池进度和下一步动作 |
| `/app/profile` | 个人资料、信仰档案、背书人提交 |
| `/app/faith-test` | 20 题测试，提交后显示通过或未通过 |
| `/app/courses` | 恋爱必修课作为必修；凯勒课作为进阶装备 |
| `/app/match` | 未入池显示锁定原因；入池后显示候选人并允许表达意向 |
| `/app/chat` | 互相表达意向后出现私聊通道，可收发消息 |
| `/app/community` | 全站广场；未入池可浏览，入池后可发帖、评论、点赞 |
| `/app/admin` | 管理员可见，包含背书审核、用户列表和基础设置 |
| `/app/vip` | 保留展示，但不影响曝光排序 |
| `/app/pastor` | MVP 暂不暴露导航入口 |
| `/app/relationships` | MVP 暂不暴露导航入口 |

前端重点不是堆入口，而是让用户清楚知道自己距离入池还差哪一步。所有被锁定的页面都应给出明确原因和可点击的下一步。

## 验收标准

使用 3 个普通账号和 1 个管理员账号验收。

| 验收项 | 通过标准 |
|---|---|
| 首页 | `/` 打开遇见路得首页，所有进入应用按钮指向 `/app` |
| 应用路由 | `/app`、`/app/login`、`/app/register` 刷新不 404 |
| 注册登录 | 普通用户可注册、登录、退出，刷新后保持会话 |
| 入池进度 | 用户能看到五项条件完成情况和缺失项 |
| 资料与信仰档案 | 保存后进度更新 |
| 信仰测试 | 通过或未通过状态影响入池资格 |
| 背书审核 | 用户提交背书，管理员审核通过后入池条件更新 |
| 恋爱必修课 | 完成后点亮入池条件，不误发凯勒课奖励 |
| 匹配 | 两个入池用户能互相看到候选人 |
| 私聊 | 双方互相表达意向后自动出现聊天通道，可收发消息 |
| 社群广场 | 未入池只能看；入池后能发帖、评论、点赞 |
| 管理台 | 普通用户访问不了，管理员能审核背书 |
| 前端构建 | `npm run build --prefix web` 通过 |
| 后端单元测试 | `node --test server/src/lib/*.test.js` 通过 |
| 数据库流 | 新库执行 schema 和 seed 后能跑通完整 MVP 闭环 |

## Git 与清理边界

实施前应先增加或确认 `.gitignore`，避免提交以下内容：

- `node_modules/`
- `.DS_Store`
- Playwright 日志与截图缓存
- 本地环境文件 `.env`
- 构建缓存和临时文件

主线文件应分批纳入版本管理。`matcha/` 体积大且为旧参考项目，是否继续保留在当前仓库主线中应单独决策，不在本 MVP 收口内处理。

## 实施顺序

1. 统一品牌与 `/app` 路由部署。
2. 修正后端 schema 和匹配/私聊断点。
3. 收口入池判断与管理员背书审核。
4. 调整 `/app` 导航和权限状态。
5. 收口社群广场基础互动。
6. 补充验收脚本、种子数据或手动验收说明。

## 成功定义

本轮完成后，遇见路得应能作为一个最小真实产品演示：用户从首页进入 `/app`，完成入池条件，通过管理员审核，进入匹配池，与另一名入池用户互相表达意向后开启私聊，并能在全站广场参与基础互动。
